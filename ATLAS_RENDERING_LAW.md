# Atlas Rendering Law

**Version:** 1.7 — Ratified (Phase 5.9 amendment: three coordinating files were never the same as one manifestation, §12)

*A one-page architecture note. Every engineer must read this before building any rendering logic for Atlas. It is not philosophy and not implementation — it is the boundary between the two.*

**The law**

Atlas does not calculate what matters. Atlas judges what matters. Every visible property in the Atlas space — position, size, clarity, prominence, warmth — is a downstream expression of that judgment. None of them is an independent decision made on its own terms.

**1. Why Atlas does not begin with a calculation**

A calculation evaluates a fact in isolation against a fixed formula and produces a comparable number by design. Atlas's judgment is not that. It interprets a fact in the context of everything else currently true about the company — the way understanding forms, not the way a score is computed. If Atlas began with a calculation, significance would already be a number before any understanding existed, and the rendering built on top of it would inherit that mechanism no matter how it was dressed up visually. A rendering engine built on a scoring formula will always regress toward being a scoring engine.

**2. Why judgment is the first step**

Nothing enters the Atlas space before Atlas has understood it. A fact becomes visible only after Atlas has formed a view on what it means in context — not because it occurred, but because Atlas judged that it matters, and could explain why. Judgment is not a filter applied after a fact is already in the space. It is the reason the fact is allowed into the space's logic at all.

**3. Why significance is a consequence, not a source**

Significance is not what Atlas computes. It is what becomes visible once judgment exists — the way light does not compute brightness, it simply makes some things visible and leaves others in shadow. Significance is never assigned to a fact directly. It is what a fact's presence in Atlas's judgment looks like, once that judgment exists.

**4. Why weight only appears at the boundary with a shared space**

Judgment on its own does not need to compare two things numerically. Comparison becomes necessary only the moment two judged things must coexist in the same limited, two-dimensional space, and one of them has to be more prominent than the other. That translation — judgment forced into a shared visual medium — is the only point where anything resembling weight is allowed to exist. Weight is not the law. It is the last, unavoidable consequence of the law meeting a shared frame.

**5. Why every future visual decision is one expression of the same judgment**

Position, size, brightness, motion, stillness — none of these are separate design decisions with their own logic. Each is a different sense through which the same underlying judgment becomes perceptible. A rendering engine should never ask, property by property, "what should this look like." It should ask once — "what has Atlas judged" — and let every visual property answer from that one source. If two properties ever disagree about what matters most right now, the rendering is wrong, not because either was badly tuned, but because they were drawn from two sources instead of one.

**6. Why a sequential narrative mode is still one judgment, not two** *(Sprint 5.4 amendment — "Jarvis Experience," Fase 5)*

Sections 1–5 describe how judgment becomes visible in a single shared space, all at once. The CEO Briefing (Fase 5) adds a second dimension: time. When Atlas narrates instead of displaying, it does not gain a second rendering law — it applies the same one across a sequence instead of a surface. Each Synthesis point is still simply the thing judgment has ranked most worth saying next; the sequence is nothing but that ranking, made perceptible one step at a time instead of many at once. A point may carry its own supporting visualisation — a highlight or emphasis on an existing Room element — but that visualisation is never a second source of truth: it is the same judgment that produced the point's words, made visible through a second sense, exactly as Principle 5 already requires of any two properties describing the same thing. A point disappears, calmly, before the next appears — not because attention decays on a timer, but because two judged things are never shown competing for the same moment any more than Principle 4 permits them to compete for the same space.

*In practice:* the sequential narrative mode is exactly one point on screen at a time, each with its own appear → pause (CEO-paced) → disappear beat, using the Room's one existing motion constant (`ROOM_MOTION.TRANSITION`) for both directions — never a second timing system. Supporting visualisation, when present, is drawn from the same existing Room materialization language, never a new rendering system of its own.

*Correction, stated openly (Phase 5.6, §7 below): this section originally closed by saying "Heart, Department Wall, and the rest of the Room's static-reveal architecture are unaffected." That claim is superseded, not silently dropped — Department Wall's permanent, always-mounted rendering was never actually outside Principle 3's reach; it was a tolerated exception. §7 removes the tolerance, not the principle.*

**7. Why Atlas is the only permanent source — the "unaffected" exception above was never a real one** *(Phase 5.6 amendment — "Atlas Space")*

§6 treated the sequential narrative mode as a special, temporal case bounded to the Executive Briefing, leaving everything else in the Room — Heart, Department Wall, the floor objects — in a separate "static-reveal" category the law didn't reach. That boundary is retired. It was never a principled one: Principle 3 already said significance is a consequence of judgment, never assigned to a fact directly, and never independent of whether Atlas currently judges it to matter. A department sitting permanently on a wall, visible regardless of whether Atlas is currently saying anything about it, was already exactly what Principle 3 forbids — a fact whose visibility traced back to nothing having ever taken it down, not to a live judgment. §6's "unaffected" carve-out didn't grant Department Wall an exemption from the law; it just left an old exception unexamined.

Atlas is the only permanent intelligent entity in the space it occupies. Every projection — a department, a graph, a timeline, a document, an Executive Briefing point — is temporary by the same reasoning every visible property always was: it exists because Atlas currently judges it worth making visible, and it stops existing the moment that judgment moves on. This is not a new rule laid on top of Principles 1–6. It is those principles, finally applied without an unexamined exception for whichever objects happened to be built first. A department does not stop existing as an organizational entity when its projection dissolves — the business model is untouched — only its claim on the shared visual space ends, exactly as Principle 4 already required of anything sharing that space with something else.

*In practice:* nothing in the Room is permanently visible except the Heart itself and the small set of navigational affordances the CEO needs to reach other parts of the space (Company Doorways, Threshold Stone, Small Hollow) — these are wayfinding, not information, and carry no business-state judgment of their own to expire. Departments, and any future category of business information, appear only for as long as Atlas is actually discussing them, via the one shared projection mechanism (`objects/DepartmentProjection.tsx`), and dissolve the moment that discussion ends — never a second, independently-tuned implementation of the same idea, per Principle 5.

*Correction, stated openly (Phase 5.7, §8 below): this section's "in practice" note treated Company Doorways, Threshold Stone and Small Hollow as permanent wayfinding *objects* — carved architecture, exempt from the law because they carry no business-state judgment. §8 finds that exemption incomplete: being exempt from judgment justified them staying visible, but never justified them staying architecture. They remain permanent as capabilities and as plain wayfinding text; the constructed objects themselves are retired.*

**8. Why Space itself is not exempt either — there is no permanent environment, only Atlas** *(Phase 5.7 amendment — "Strip Space to Atlas")*

§7 retired the exception for permanent *business-state* objects (Department Wall) but left the Room's own architecture — a bounded stage frame, a wall plane, a floor plane, carved recesses and inlays — outside the law entirely, on the reasoning that architecture isn't a judgment at all, just a place for judgments to appear. That reasoning is incomplete, not wrong: a wall and a floor genuinely carry no business-state judgment, but they are still a permanent visual claim on the shared space — the exact thing Principle 4 says may only exist as "the last, unavoidable consequence of the law meeting a shared frame," never as a precondition the frame arrives with already built. A constructed room is a shared frame that existed before any judgment did. Atlas was never meant to stand inside one.

Atlas is the only permanent presence in Space. Not the Heart-as-object among other objects — Atlas, full stop; Space itself carries no permanent business meaning and asserts no architecture of its own. A visual exists only while Atlas needs it to communicate, whether that visual is a department, a document, a graph, or the very ground it seems to stand on. Wayfinding — the CEO's ability to reach CEO Inbox, AI Tools, and the two companies — is preserved as a capability and as restrained, plain text (`RoomScreen.tsx`'s wayfinding row), because reaching them is a real, permanent need; but the carved stone objects that used to represent that need are not preserved, because a carved object is architecture, and architecture is exactly what this section retires.

*In practice:* the only thing permanently mounted in Space is the Heart. Its glow, scale and presence still express one continuous judgment (`heartVitality`, `activated`) exactly as Principles 1–5 already require. Everything else — a department's projection, the Executive Briefing itself, a Company Interior's own ambient field — exists only while Atlas is actively using it to communicate, and Space returns to empty the moment it isn't. This is not a stricter rule bolted onto §7; it is §7's own reasoning, extended to the one category it hadn't yet reached.

*Correction, stated openly (Phase 5.7b, §9 below): this section said wayfinding text "is preserved as a capability and as restrained, plain text... permanently." That was still one exception too many. §9 removes it — not because the reasoning about architecture-versus-capability was wrong, but because "permanent" was never actually required by that reasoning; it was a leftover habit from when the objects it replaced were permanent.*

**9. Why there is no principled floor beneath "Atlas is the only permanent thing"** *(Phase 5.7b amendment — "Atlas Is The Product")*

§8 drew a line between permanent business-state architecture (retired) and permanent wayfinding text (kept, on the reasoning that reaching CEO Inbox, AI Tools and the two companies is a real, ongoing need). That line was already inconsistent with §8's own opening sentence: "Atlas is the only permanent presence in Space." A real, ongoing need for a *capability* does not require a permanent visual claim on Space to satisfy it — it requires only that the capability be reachable when it is actually wanted, which is a different, weaker requirement §8 conflated with permanence out of habit, not out of principle. There is no version of "Atlas is the only permanent thing" that also permits a second permanent thing, however small or however plainly it's written.

Wayfinding text now appears only while Atlas is activated (`RoomScreen.tsx`), the same lifecycle the Executive Briefing already uses, rather than sitting in Space unconditionally. This is not a new mechanism — it reuses the one activation state Space already has — and it is not a capability loss: CEO Inbox, AI Tools and both companies remain exactly as reachable as before, one tap later. What changes is only that reaching them is now itself something Atlas's presence enables, rather than something the empty Space offered on its own before Atlas was ever engaged.

*In practice:* at rest, Space contains exactly one thing: Atlas. No wall, no floor, no department, no wayfinding, no chrome. The one necessary exception — a single plain-text link to leave the screen entirely — is not part of Space's own materialization at all; it is the ordinary, unavoidable minimum every screen in the application needs to be exitable, the same category as a browser's own back button, not a "Room" affordance this law has ever governed.

*Correction, stated openly (Phase 5.7.1, §10 below): this section removed wayfinding's permanence but kept it visible, activated, as plain text. §10 finds that distinction didn't hold — visibility, not permanence, was the actual thing Principle 3 forbids a fact having independent of Atlas's own judgment. Text with no background is still a menu; being gated to `activated` made it temporary, not un-navigational.*

**10. Why "not permanent" was not the same test as "not navigation"** *(Phase 5.7.1 amendment — "Remove Visible Navigation From Space")*

§9 correctly removed wayfinding's permanence, gating it to `activated` so it no longer sat in Space unconditionally. But that fix answered the wrong question. Principles 1–5 were never about how long something stays visible — a fact appearing for one second on Atlas's own judgment is exactly as licensed as one appearing for an hour on Atlas's own judgment. What they forbid is a visible property whose presence traces back to something other than Atlas currently judging it worth showing. The wayfinding row's presence never traced back to a judgment at all; it traced back to the CEO having activated Atlas, which is a lifecycle event, not Atlas deciding CEO Inbox is worth mentioning right now. Gating it to `activated` disguised an ungoverned menu as a governed one by attaching it to the right variable for the wrong reason.

CEO Inbox, AI Tools and the two Company Doorways remain real, reachable capabilities — nothing about routing, `handleSelect`, or the underlying business model changes here. What changes is that reaching them is no longer something Space itself ever offers as a visible option, at any point in its lifecycle. Reaching them today, before Atlas can be asked in plain language, is development infrastructure (`RoomScreen.tsx`'s `handleDevNavigation`, gated behind `__DEV__` and triggered by an unlabeled hit region rather than a visible control), not a second, quieter navigation system living alongside Atlas.

*In practice:* Space contains no visible navigation at all, activated or not — not a bar, not plain text, not an icon row. The only thing a CEO can ever see in Space is Atlas itself and the one unavoidable exit link every screen needs. A production build contains zero code paths that could ever show CEO Inbox, AI Tools, or either company as a persistent option; a development build contains exactly one, and it is deliberately built to look like what it is — temporary developer infrastructure — rather than like a preview of a future design.

*Correction, stated openly (Phase 5.8, §11 below): this section still described Space as a container Atlas is displayed inside — "the only thing a CEO can ever see in Space" — with an exit link accepted as the one legitimate resident of that container besides Atlas. §11 finds both the container framing and the exception it produced were never necessary.*

**11. Why Atlas is the manifestation, not a scene it stands inside — and why even the exit link was never necessary** *(Phase 5.8 amendment — "Rebuild Space Around Atlas")*

Every section above, including §10, still reasoned about what is allowed to exist *in* Space, as if Space were a container with rules about its contents and Atlas were its most-favored resident. That framing was the last unexamined assumption, and this phase's own review named it directly: "Atlas does not live inside an interface. Atlas IS the interface." A container that happens to be empty except for Atlas is still a container. The correct picture was never "Space, containing only Atlas" — it was "Atlas, whose existence is what Space's appearance *is*." Nothing about that distinction changes what a CEO sees. It changes what the code is allowed to be structured as: not a scene component that composes Atlas alongside other layers (`RoomScene.tsx`, renamed `AtlasManifestation.tsx` this phase), and not a Briefing mounted as an independent sibling overlay near Atlas (`ExecutiveBriefingOverlay.tsx`, renamed `AtlasBriefingExpression.tsx`), but one continuous manifestation in which a Briefing appearing and Atlas brightening are two visible consequences of the same one thing.

This phase's review also refined what Principles 1–5 already implied but had never been forced to say explicitly: Atlas does not have modes. "Atlas briefs," "Atlas reveals a department," "Atlas is activated" all still describe Atlas as entering named, switchable states — better than "BriefingOverlay is visible," but not yet the actual claim, which is that these are not states Atlas enters at all, only names given afterward to a stretch of one continuous, undifferentiated presence. Internal state remains necessary and is not banned by this — `RoomScreen.tsx` still holds a real `activated` boolean, `AtlasBriefingExpression.tsx` still holds a real step index — but this phase collapses what used to be two booleans kept in lockstep (`activated` and `briefingOpen`, the second one an effect syncing the first) into one, because the two were never actually two different facts, only two variables narrating the same one.

The ambient light around Atlas (`vignette`, `ambientCore` in `AtlasManifestation.tsx`) is corrected the same way: previously a flat, constant-opacity layer, present at the same intensity regardless of anything Atlas was doing — independent scenery, not something Atlas casts. Its intensity is now a direct function of `heartVitality`, the same real signal Heart's own rings already read, so the atmosphere is legible as a byproduct of Atlas existing rather than a backdrop with its own independent brightness.

Finally, the exit link. §8 and §9 both treated it as the one necessary exception to "Atlas is the only permanent thing" — "the ordinary, unavoidable minimum every screen needs." That reasoning didn't survive this phase's own brief, which lists "visible exit links" and "visible back links" explicitly among the legacy chrome to remove and states plainly that "nothing related to navigation may be visible in the production experience." The exit link was never actually unavoidable; it was a convenience carried over from treating Space as a screen with the usual screen affordances. It is removed this phase. Leaving Atlas Space now happens through the browser or OS's own back affordance, or through `handleDevNavigation`'s new "← Back" option — development infrastructure, not a residents' list for the container Space no longer is.

*In practice:* Atlas is not rendered inside a scene, a stage, or a container with an exception list. There is one manifestation; Atlas's own presence is what that manifestation is made of. No other component in `src/atlas/studio/room/` mounts as an independent visible surface with its own backdrop, card, or dismiss chrome — the two that used to (`RoomScene.tsx`, `ExecutiveBriefingOverlay.tsx`) no longer exist under those names or in that shape. `CeoFocusOverlay.tsx`, `PlaceholderOverlay.tsx` and `CompanyInterior.tsx` still do, unchanged — this phase's brief scoped the rebuild to Atlas's own primary rest/activation/briefing experience, and those three are reachable today only through `handleDevNavigation`'s development-only mechanism, not through anything Space itself ever shows.

*Correction, stated openly (Phase 5.9, §12 below): §11 corrected what Space was — a manifestation, not a container — but left the *code* structured as three files (`RoomScreen.tsx`, `AtlasManifestation.tsx`, `AtlasBriefingExpression.tsx`) coordinating through props to produce that one manifestation. §12 finds that three components agreeing to look like one thing is not the same as one thing.*

**12. Why three coordinating files were never the same as one manifestation** *(Phase 5.9 amendment — "Complete Presentation Reset")*

§11 renamed the scene component and the Briefing overlay and merged their rendering into a shared visual anchor, and that was real progress — but the underlying shape was unchanged: three separate files, three separate sets of props, kept in sync by one passing state down to the next. A screen owned `activated`; a manifestation component received it and derived glow styles from it; a briefing-expression component received a further-derived `active` flag and its own callback back up to the screen. That is still coordination between independent parts, just quieter coordination than before. Nothing in Principles 1–5 forbids multiple files — `Heart.tsx` and `DepartmentProjection.tsx` are real, separate, and correctly so, because they are self-contained visual leaves with no state of their own to keep in sync with anything. What §12 retires is coordination for its own sake: the screen, the void, Atlas's core's presence, and Atlas's briefing content collapsing into `AtlasSpace.tsx`, one component with one set of state, because they were never actually three things agreeing — they were one thing that had been organized as three.

This phase's brief was explicit that reuse should not be preferred merely because it already exists and typechecks. `RoomScreen.tsx`, `AtlasManifestation.tsx`, and `AtlasBriefingExpression.tsx` are deleted, not renamed a third time. `AtlasSpace.tsx` is a new file, written from nothing, that happens to call the same `roomData.ts`/`briefingSteps.ts`/`useControlDashboard()` functions those three files called, because that is real business logic this sprint does not touch — reusing a function call is not reusing a component tree. `Heart.tsx` and `DepartmentProjection.tsx` keep their file paths (`CompanyInterior.tsx`, out of this sprint's scope, imports `Heart.tsx` directly) but their contents are likewise rewritten from a blank file, not edited from the previous version.

*In practice:* the presentation layer for Atlas's own primary experience is now two files, not four: `AtlasSpace.tsx` (the whole visible experience — void, core presence, briefing expression, one state) and `objects/Heart.tsx` (the one self-contained visual leaf reused by both `AtlasSpace.tsx` and `CompanyInterior.tsx`, exactly the kind of genuine, self-contained reuse Principle 5 never forbade). `DepartmentProjection.tsx` remains a third, smaller leaf for the one non-text visual `AtlasSpace.tsx`'s briefing content produces. Fewer files was never the goal in itself; it is the visible consequence of there being one manifestation instead of several components narrating one on each other's behalf.

**The principle this protects**

Every visual expression of Atlas must trace back to a single judgment. There can be many expressions of it — position, size, clarity, weight, stillness — but never two sources. The moment two visual properties require two different truths to both appear correct, the rendering has stopped representing Atlas and started representing competing systems wearing the same frame.
